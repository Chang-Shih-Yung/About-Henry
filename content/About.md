## Q：在開發過程中你遇過什麼挑戰？你怎麼應對？

**A：**

**Situation｜情境**
在接手 MES 智慧工廠系統時，原有的報工流程是由多個老模組拼接而成，現場作業員在報工操作時錯誤頻繁且繁瑣。為了不中斷產線，所以必須在不中斷生產的前提下重構。

**Task｜任務**
我的任務是重構報工操作介面與流程體案——但不能停線，後端與資料庫繼續跑，前端同時更新，讓現場作業員不能感覺到我們「正在換系統」。

**Action｜行動**
- 第一步我跟SA先把工單的狀態資訊架構、功能清單梳理清楚。
- 舊系統的判斷邏輯散落在各個元件裡，頁面中太多重複的功能觸發點，誇張一點一個頁面同時有20個按鈕，所以我先把所有狀態集中管理、刪減重複的流程。
- 介面上也跑了幾輪跨部門 Key User 訪談，把他們說不清楚、但明顯的操作習慣，轉成具體的 UI 引導。

**Result｜結果**
- 現場報工錯誤率下降 **33%**、整體作業效率提升 **75%**。
- 這個專案後來榮獲 SAP「馬達雲」**最佳創新獎**。

## Q：發覺流程痛點

**A：**
從這個案例我學到：客戶通常不會主動告訴你他真正要什麼——因為他比你更不懂這個領域，他無法精確描述細節。真正的挑戰不是技術難度，而是**把隱性的痛點挖出來，然後用系統設計解決它**。

1. 觀察使用者行為（而不是只聽需求）
現場人員通常用很口語的方式描述操作，但那其實是「既有系統的使用習慣」，不一定是最佳流程。
2. 將行為抽象成流程
把「點來點去」整理成明確流程節點（查詢 → 確認 → 填寫 → 再確認），找出實際操作邏輯。
3. 找出痛點（斷點與重複）
例如：
	•	資料重複查詢
	•	頁面來回跳轉
	•	資訊分散導致判斷成本高

4.進行流程重構（UI/UX 解法）
	•	單一入口（Single Entry）
	•	關鍵資訊前置
	•	流程線性化（減少跳頁）
	•	5️⃣ 核心思維
使用者講的是「怎麼做」，但我要找的是「為什麼這樣做」，再用系統設計把流程簡化。

## Q：你說有把「現代化 UI 與體驗」融入系統，這對工廠系統來說具體是什麼？

**A：**
無論第一代的工廠或是業務系統，都有共同的痛點，就是：密密麻麻的表單、四散的按鈕、未整理的狀態、模糊的卡控，以及介面停留在十年前的視覺語言。「現代化」對我來說不是追流行，而是把 **使用者負擔降到最低**。

幾個實際做過的例子：

- **即時前置驗證**：操作離開欄位的瞬間就提示警告，不等送出。
- **狀態視覺化**：因為工廠現場吵、昏暗、沒時間讀文字——所以流程大多以高對比的 狀態呈現，讓作業員掃一眼就能判斷該做什麼操作。
- **步驟式引導取代無限scroll的長表單**：降低填錯的機率與即時提示

所以以上經驗成為我開始開發無障礙元件庫的動機：

- **無障礙 UI 元件**：我主導企業級元件庫的建立，主要是：參考知名開源元件庫，綜合前面兩個系統的經驗，這不只是「符合法規」，而是希望讓不同使用情境（鍵盤操作者、螢幕放大、工廠現場戴手套觸控）都能順暢使用。
- **OKLCH 顏色系統**：相較傳統 HEX/RGB ，更貼近肉眼判讀的顏色。

這些改動在技術上不算難，但對在高壓、快節奏的現場環境工作的作業員來說，**少一次誤觸、少一個彈窗，就是少一次出錯的機會**。

## Q：你怎麼理解「流程設計」對使用者的價值？

**A：**
- 當我們認為某個操作流程是常識，往往是因為我們已經太熟悉了。但對使用者來說，能讓他們第一次接觸到「不需要思考就能順利走完」的流程，其實就是一個新鮮又能建立信任感的亮點。

- 流程做得好，使用者不會說「這很好用」——他們只會自然地繼續用下去，這才是真正的成功。就像真正好吃的東西，是在不知不覺中就把它吃完了。

## Q：我如何保持技術上的持續成長？

**A：**
我蠻習慣把學習融入日常的。下班後如果有空，通常會做一些 side project 或探索新技術，不一定是為了馬上用到，而是讓腦袋先有個印象，等真正需要時能更快切入。

AI 工具的普及也改變了我的學習節奏。以前學東西習慣從底層了解每個知識點，現在反而會先跑一個粗糙的 prototype，再從中去了解用了哪些技術，我也比以前更敢嘗試一些「太花時間」的技術。

舉個最近的例子：我在準備這次面試的空檔，用 Vue 框架實作了一個雲端筆記本，搭配 Vercel 部署與 GitHub API 作為資料儲存，讓自己在出國期間也能隨時用手機複習面試內容、並即時修改筆記。從發想到上線大概花了一個週末——這對我來說就是「學習融入日常」最直接的體現。
