## Q：在開發過程中你遇過什麼挑戰？你怎麼應對？

**A：**

主要有兩個：

**第一個是「客戶說不清楚需求」。**
現場人員通常是用口語描述操作習慣，但那不代表是最佳流程。
所以我不會只聽需求，我會直接去觀察他們怎麼操作——

比如我發現作業員在報工時，要在工單選擇、數量輸入、狀態確認之間來回跳，這不是他們不會用，是資訊分散、狀態不清楚。
我把這段流程拆解後，做了幾件事：
- 把關鍵欄位集中在同一個畫面
- 加入預設值，減少重複輸入
- 操作完立刻給即時回饋（成功 / 異常提示）
- 把流程改成線性的一步一步引導

還有一個延伸的做法是：因為工廠現場很吵、光線也不穩定，所以我把狀態改成高對比的視覺呈現，讓作業員掃一眼就能判斷。這讓操作錯誤率明顯降低，也是我後來主導建立企業元件庫的起點。

**第二個是「既有系統太老舊、但不能停機重構」。**
老系統通常是表單複雜、狀態模糊、模組東拼西湊。
我們的做法是先梳理核心流程，再分階段替換，同時保持舊系統平行運作，不中斷現場作業。

整體下來我學到一件事：真正的挑戰通常不在技術，而在於**怎麼把隱性的痛點找出來，再用系統設計把它具象化、解掉**。

## Q：你怎麼理解「流程設計」對使用者的價值？

**A：**
- 當我們覺得某個操作是「常識」，通常是因為我們太熟了。但對使用者來說，能讓他們第一次接觸就不需要思考、自然走完，這才是真正做到位。

- 流程做得好，使用者不會說「這很好用」——他們只會自然地繼續用下去。就像好吃的東西，是在不知不覺中就把它吃完了。

## Q：我如何保持技術上的持續成長？

**A：**
我蠻習慣把學習融入日常的。下班後如果有空，通常會做一些 side project 或探索新技術，不一定是為了馬上用到，而是讓腦袋先有個印象，等真正需要時能更快切入。

AI 工具的普及也改變了我的學習節奏。以前學東西習慣從底層了解每個知識點，現在反而會先跑一個粗糙的 prototype，再從中去了解用了哪些技術，我也比以前更敢嘗試一些「太花時間」的技術。

舉個最近的例子：我在準備這次面試的空檔，用 Vue 框架實作了一個雲端筆記本，搭配 Vercel 部署與 GitHub API 作為資料儲存，讓自己在出國期間也能隨時用手機複習面試內容、並即時修改筆記。從發想到上線大概花了一個週末——這對我來說就是「學習融入日常」最直接的體現。
