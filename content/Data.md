## Q1：你有沒有分析過使用者行為或數位軌跡的經驗？

**A：**

有，雖然我的主力是前端，但分析用戶行為一直是我工作的一部分：

- 在 **MES 智慧工廠**專案，我會去追蹤現場工人的操作路徑——哪個步驟最多人出錯、哪裡停留時間最長，然後拿這些資料來重構報工流程，最後錯誤率降了 33%

---

## Q2：什麼是「數據中台」？你怎麼理解這個概念？

**A：**

簡單說就是把數據的「收集、整理、提供」集中管理的一個平台：

- 數據中台的目標是：**一個地方統一管數據，其他人都來這邊取用**
- 對我來說，這跟前端的「狀態管理」概念很像——把分散的狀態集中到 store，其他 component 統一從 store 取，不自己亂存

---

## Q3：你說的「優化導客路徑」，從前端角度具體怎麼做？

**A：**

我前端有幾個實際做法：

- **第一步：分析用戶卡點**：先看哪個頁面流失率高、哪個操作最多人放棄或點擊返回，或是回應的錯誤代碼最多，然後去定義出問題點。
- **第二步：簡化流程步驟**：比如在 MES 我把多步驟報工壓縮，並加前置驗證，讓用戶在送出前就能知道哪邊填錯了
- **第三步：狀態回饋**：用戶每個動作都要有反應——loading、成功、失敗都要清楚告知，不讓用戶猜

這當然是跟後端協作的結果、最後也成功加速了工廠人員的操作效率

---

## Q4：怎麼監控前端跟數據服務的性能？

**A：**

我主要從這幾個維度看：

- **前端層**：監控 API response time、頁面載入時間（LCP、FID、CLS），用 browser DevTools 或接 Sentry 這類工具
- **API 層**：觀察 error rate、timeout 頻率，如果某支 API 特別慢就要往下追是 DB 問題還是查詢邏輯問題
- **CI/CD 整合**：我有整合過 AI Agent 在 pipeline 裡跑自動化檢查，出問題自動通知
- 我的角色通常是：**前端端先發現異常，再反向協助後端定位問題**，跨角色協作是我的強項之一

## Q5：你怎麼跟數據分析師、後端、業務部門合作？

**A：**

這在 EQS 全球報價系統有很完整的實戰：

- **跟業務部門**：他們提需求，但需求往往不精確——我會把模糊的業務語言轉換成具體的流程圖，再回去確認，避免做完才說不對
- **跟系統分析師**：一起定義簽核規則與卡控條件，把複雜邏輯模組化，讓前端可以統一調用，不用每個地方手寫 if-else
- **跟後端**：討論 API 規格，確保數據格式前後端對齊，減少來回
- 核心原則：**我是翻譯的角色**——把技術語言翻成業務語言，把業務需求翻成技術規格，讓不同背景的人能對話

---

## Q6：你有建立過技術文件或使用指南嗎？

**A：**

有，這一直是我工作的一部分：

- 在 MES 和 EQS 專案，我都有寫過給現場人員或業務人員看的**使用說明**，重點是：要站在「不懂技術的人」的角度寫，不能用工程師語言
- 技術文件我會寫 **API 規格文件**（哪支 API、傳什麼參數、回什麼格式）和**流程說明**（狀態機的轉換邏輯）
- 我理解文件的價值不是給自己看的——是讓後繼的人能快速上手，減少重複溝通成本
- 這次準備面試也有自己架了這個 [關於 Henry](https://about-henry.vercel.app) 作為展示用途，某種程度也是一種文件化的思路

## Q7：你沒有數據中台的直接開發經驗，你怎麼補這個缺口？

**A：**

坦白說，我沒有直接開發過數據中台的後端架構，但我有幾個切入點：

- **我懂前端消費數據的痛點**：API 格式不統一、數據延遲、一致性問題，這些我都踩過，所以我知道「中台需要解決什麼問題」。當然我知道企業對中台的定義都不一樣，所以我還是會先觀察學習、然後才實作。
- **我有跨系統整合的經驗**：MES 整合 SAP 等多個資料來源，這跟數據中台的「統一數據源」邏輯相通
- **我學得快**：我從 UI/UX 轉前端工程，再整合 CI/CD 跟 AI Agent，都是邊做邊學的——新的技術棧我能快速上手
- **我能補上前端視角**：數據中台最終產出是要讓人用的，這塊我有很強的使用者思維
