## Q：在開發過程中你遇過什麼挑戰？你怎麼應對？

**A：**

**Situation｜情境**
在接手 MES 智慧工廠系統時，原有的報工流程是由多個老模組拼接而成，現場作業員在填報工單時錯誤頻繁，錯誤的資料會直接影響後段的生產排程與績效計算，但又不能停線大改，必須在不中斷生產的前提下重構。

**Task｜任務**
我的任務是以 Angular 19+ 重新設計報工核心流程與工單機制，同時讓新舊系統可以並行過渡，不影響現場運作。

**Action｜行動**
我從「狀態治理」切入：梳理工單所有可能的狀態流，建立明確的狀態機，並在每個關鍵節點加入前置驗證——讓系統在資料「即將出錯」的那一刻就攔截，而不是等到下游報錯才追溯。同時整合跨部門 Key User 的回饋，把隱性的操作習慣轉化成顯性的 UI 引導。

**Result｜結果**
現場報工錯誤率下降 **33%**、整體作業效率提升 **75%**。這個專案後來榮獲 SAP「馬達雲」**最佳創新獎**。

---

從這個案例我學到：客戶通常不會主動告訴你他真正要什麼——因為他比你更不懂這個領域，他無法精確描述細節。真正的挑戰不是技術難度，而是**把隱性的痛點挖出來，然後用系統設計解決它**。

## Q：你說有把「現代化 UI 與體驗」融入系統，這對工廠系統來說具體是什麼？

**A：**
工廠系統傳統上長這樣：密密麻麻的表單、全頁跳轉、送出才報錯，介面停留在十年前的視覺語言。「現代化」對我來說不是追流行，而是把 **使用者負擔降到最低**。

幾個實際做過的例子：

- **即時前置驗證**：工單數量填超出範圍，游標離開欄位的瞬間就提示，不等送出。過去的設計是送出後才彈 alert，現場作業員已經按了下一步才被打斷，現在攔在輸入當下。
- **狀態機視覺化**：工單有「待確認 / 執行中 / 暫停 / 完工」四種狀態，舊系統用文字顯示，改成 **色彩 badge + 明確的可操作按鈕**，讓現場人員一眼判斷能做什麼、不能做什麼。
- **步驟式引導取代長表單**：把原本一頁十幾個欄位的報工表單，拆成 3 步驟的 Stepper——每步只問必要的資訊，降低填錯的機率。
- **非阻斷式回饋**：成功/失敗改用 Toast 提示，不用 alert 彈窗打斷操作節奏，現場手不離鍵盤也能感知系統狀態。

除此之外，現代化對我來說也包含兩個更底層的事情：

- **無障礙 UI 元件**：我主導過企業級元件庫的建立，所有元件都有正確的 ARIA 標籤、keyboard navigation、focus management。這不只是「符合法規」，而是讓不同使用情境（鍵盤操作者、螢幕放大、工廠現場戴手套觸控）都能順暢使用。
- **OKLCH 顏色系統**：傳統 HEX/RGB 在做深色模式、主題切換時容易出現顏色不一致或對比度不足的問題。改用 OKLCH 之後，顏色在視覺感知上是線性的——調亮度不會讓色相偏移，深色模式的對比自動維持，整個 Design Token 系統也更容易維護。

這些改動在技術上不算難，但對在高壓、快節奏的現場環境工作的作業員來說，**少一次誤觸、少一個彈窗，就是少一次出錯的機會**。

## Q：你怎麼理解「流程設計」對使用者的價值？

**A：**
當我們認為某個操作流程是常識，往往是因為我們已經太熟悉了。但對使用者來說，第一次接觸到一個「連我都不需要想就能走完」的流程，恰好是個既新鮮又能建立信任感的賣點。

流程做得好，使用者不會說「這很好用」——他們只會自然地繼續用下去，這才是真正的成功。

## Q：我如何保持技術上的持續成長？

**A：**
我蠻習慣把學習融入日常的。下班後如果有空，通常會做一些 side project 或探索新技術，不一定是為了馬上用到，而是讓腦袋先有個印象，等真正需要時能更快切入。

這幾年 AI 工具的興起，也讓我在技術嘗試上可以更快驗證想法——以前可能要花一週搭起來的概念，現在可能只需要一個下午。這讓我在選擇技術方向時更敢嘗試、也更能提早判斷值不值得深入。

舉個最近的例子：我在準備這次面試的空檔，用 Vue 框架實作了一個雲端筆記本，搭配 Vercel 部署與 GitHub API 作為資料儲存，讓自己在出國期間也能隨時用手機複習面試內容、並即時修改筆記。從發想到上線大概花了一個週末——這對我來說就是「學習融入日常」最直接的體現。
